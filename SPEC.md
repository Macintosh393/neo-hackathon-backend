# Product Specification: AI Time-Manager (SPEC.md)


## 1. Product Vision & Overview
Name: AI Time-Manager
Target Audience: Software Engineering and IT students.
Core Problem: Students struggle with time management, procrastination, and decomposing complex projects (like writing a backend or preparing for a database exam) into manageable study sessions, especially when balancing personal lives and university schedules.
Solution: An automated, AI-driven scheduling backend. The system ingests academic deadlines, analyzes the user's personal capacity, cross-references real-time availability via Google Calendar, and uses an LLM (Google Gemini) to generate and schedule an optimal, conflict-free timeline of study sessions.

## 2. Final State & MVP Scope
1. The final MVP is a headless Node.js API supporting a modern frontend client. It must deliver the following core capabilities:
2. Frictionless Onboarding: Google OAuth integration with offline access for background syncing.
3. Persona Management: Storing user constraints (max study hours per day, preferred time of day, weekend availability).
4. AI Decomposition: Parsing unstructured project data (via JSON import) and using LLMs to break large academic tasks into 1-2 hour focused study sessions.
5. Smart Scheduling: A mathematical algorithm that finds available time blocks in the user's real-life schedule (Google Calendar) and safely places AI-generated sessions without exceeding personal limits.
6. Aggregated Dashboard: Real-time calculation of overall progress, upcoming deadlines, and daily agenda without performance bottlenecks.
7. Autopilot (Nightly Cron): Background jobs that automatically detect missed study sessions and reschedule them to future available slots.

## 3. Domain Model & Entities
- The system uses a relational database (PostgreSQL via Prisma).
- User: The central entity. Holds authentication data, Google OAuth refresh_token, and the persona JSON (which dictates the scheduling constraints).
- Project: Represents a high-level academic goal (e.g., "Coursework for OS", "Build a REST API"). Has a strict deadline and belongs to a specific courseName.
- StudySession: The atomic unit of work generated by the AI. Represents a specific task (e.g., "Design database schema"). It has a startTime, endTime, durationMinutes, and a status (SCHEDULED, COMPLETED, MISSED). A StudySession belongs to a Project.

## 4. High-Level Architecture
- The application is built on Node.js + Express using a strict Feature-Layered Architecture (Separation of Concerns):
- Transport Layer (Controllers & Routes): Exclusively handles HTTP requests, responses, and payload validation (using celebrate/Joi).
- Service Layer (Business Logic): Pure functions and classes handling the core app logic. Completely isolated from HTTP objects (req/res).
- Data Layer (Prisma): Handles all database interactions. Complex metrics are aggregated here using native DB functions, not calculated in memory.
- External Adapters: Isolated services for communicating with Google Calendar API and Google Gemini API.

## 5. Core Business Logic & Mechanics
### 5.1. Multi-Calendar Free/Busy Resolution
- The system must never double-book a user.
- It fetches a list of all user-owned calendars via Google API.
- It filters out non-actionable calendars (like national holidays).
- It queries the freebusy endpoint across all relevant calendars to generate a unified array of occupied time slots.
- Privacy Rule: The system NEVER fetches or stores the titles, descriptions, or attendees of personal Google Calendar events. It only reads time boundaries.

### 5.2. AI-Driven Project Breakdown
- When a user uploads projects, the AiAdapter queries Google Gemini.
- The AI is instructed to return a strictly typed JSON array of logical steps.
- The AI does not assign dates or times (that is the mathematical scheduler's job). It only determines the title and estimated duration of each session.

### 5.3. The Greedy Scheduler Algorithm
- This is the mathematical heart of the application.
- Inputs: Generated sessions, User Persona, Unified Google Calendar 'busy' slots, Project Deadline.
- Constraints applied sequentially:
- Skip days if studyOnWeekends is false.
- Restrict placement to the user's preferredTime window (Morning, Afternoon, Evening).
- Ensure the sum of study session durations on any single day never exceeds maxHoursPerDay.
- Ensure sessions never overlap with Google Calendar 'busy' slots.
- Fallback: If a project mathematically cannot fit before the deadline given the constraints, the system must handle it gracefully (e.g., compress sessions, bypass preferred time, or return a warning flag) rather than crashing or infinite looping.

### 5.4. Dashboard Data Aggregation
- The Dashboard requires complex metrics (completion percentages, days left, grouped progress by course).
- Performance Rule: This must be solved using database aggregations (e.g., Prisma groupBy, _count, SUM) or raw SQL queries. Fetching all projects and sessions into the Node.js event loop to calculate percentages using .map() and .reduce() is strictly prohibited.

### 5.5. Nightly Rescheduler (Autopilot)
- A background worker (node-cron) runs every night at 03:00.
- It identifies any StudySession with a startTime in the past that is still marked as SCHEDULED.
- It marks them as MISSED.
- It uses the stored googleRefreshToken to fetch fresh calendar data.
- It invokes the Greedy Scheduler to dynamically shift these missed sessions into future available time slots, adjusting the entire timeline automatically.
